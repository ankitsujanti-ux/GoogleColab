Timeline

  1. 2017 — Google: paper “Attention Is All You Need” — new way to wire neurons (architecture) with self-attention: learn which parts of an input sequence matter.
  2. 2018 — OpenAI GPT-1 — early, basic.
  3. 2019 — GPT-2 — more notice in the field.
  4. 2020 — GPT-3 — scale jump; “just statistics” vs “something huge” debate.
  5. 2022 — ChatGPT (GPT-3.5 + chat/RLHF-style alignment) — chat UX exploded adoption.
  6. 2023+ — GPT-4, multimodal (4o), GPT-5, etc.

From traditional ML to deep learning (brushstrokes)

  • Traditional ML: statistical model; learn parameters from labeled examples (e.g. credit score from features).
  • Neural network (~1950s idea): many small “neurons” connected — loosely inspired by brains, not a brain.
  • Deep learning: many stacked layers → can learn richer patterns with more data and compute.
  • 2017 breakthrough: transformer architecture + attention → scale to bigger data & models more efficiently.
Key mindset: The transformer is a clever efficiency optimization for training and inference at scale — not proven to be the only possible path to intelligence. Alternatives exist (state-space models, hybrids); none has clearly replaced transformers for mainstream LLMs yet. Without transformers, progress might have been slower and API costs potentially much higher.

Tokens (notebook)

Models do not read “words” — they read token IDs from a vocabulary. Same word can split into multiple tokens; rare words often split more.

import tiktoken
encoding = tiktoken.encoding_for_model("gpt-4.1-mini")
tokens = encoding.encode("Hi my name is Ed and I like banoffee pie")
# Decode one token at a time to see subword pieces
  • Encode: text → list of integer token IDs.
  • Decode: token ID(s) → text fragment.
  • Why care: billing, context limits, and prompt design are all token-based.

Stateless API & the “memory” illusion (notebook)

Messages format

Chat API expects a list of dicts with role and content:

messages = [
    {"role": "system", "content": "You are a helpful assistant"},
    {"role": "user", "content": "Hi! I'm Ed!"}
]
response = openai.chat.completions.create(model="gpt-4.1-mini", messages=messages)

The surprise

New call with only "What's my name?" → model does not know you said “Ed” before.

Why

Every API call is stateless — a fresh forward pass. No server-side chat memory unless you send history.

The fix (what ChatGPT does)

messages = [
    {"role": "system", "content": "You are a helpful assistant"},
    {"role": "user", "content": "Hi! I'm Ed!"},
    {"role": "assistant", "content": "Hi Ed! How can I assist you today?"},
    {"role": "user", "content": "What's my name?"}
]

Model sees full sequence → predicts likely next tokens → “Ed” appears because it’s in the context.

Five rules to remember (notebook recap)

  1. Every LLM call is stateless.
  2. You pass the entire conversation each time (when you want continuity).
  3. That creates the illusion of memory.
  4. It’s not recall — it’s next-token prediction over whatever text you included.
  5. Longer history → more tokens → higher cost — and that’s intentional (more context = more compute).

Bridges to later topics

  • Context window: max tokens in one request — design prompts and history trimming around this.
  • Context engineering: what you put in the message list (system, tools, RAG chunks).
  • Agent loops: repeated stateless calls + tool results appended to messages.
  • Parameters / temperature: control randomness of token sampling (later labs).

Teach someone else — 60-second version

GPT is a pre-trained transformer that predicts the next token. Transformers won because attention lets them scale efficiently on sequences. When you chat with an API, nothing is remembered unless you resend the full thread — tokens are how the model reads text and how you pay. Memory in products is engineered, not built into a single call.
